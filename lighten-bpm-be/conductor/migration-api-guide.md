## 1. Concepts

### Identity — how entities are matched across environments

| Entity | Stable key used |
|---|---|
| Form, Workflow, Revision | `public_id` (12-char alphanumeric string — see below) |
| Validation Registry rule | `name` (human-readable string) |
| Tag | `name` |
| Master Data dataset | `name` |
| User | `code` |
| Org Unit / Role | `code` |

#### `public_id` format

`public_id` values are generated by the server using [nanoid](https://github.com/ai/nanoid) with a custom alphabet of `0-9A-Za-z` (no hyphens, no underscores). The random part is always **12 characters**. An environment-level prefix is optionally prepended via the `PUBLIC_ID_PREFIX` env var, making the total length `prefix.length + 12`.

| Deployment | `PUBLIC_ID_PREFIX` | Example `public_id` |
|---|---|---|
| No prefix (default) | *(empty)* | `Xk7mN3pQr8sT` |
| Dev environment | `DEV` | `DEVXk7mN3pQr8sT` |
| Production environment | `PRD` | `PRDXk7mN3pQr8sT` |

The prefix is the mechanism that guarantees IDs generated in different environments never collide. A `public_id` that starts with `DEV` could never be generated in a `PRD`-prefixed environment.

> This format is **not** a UUID. Do not generate or validate `public_id` values against UUID patterns on the client side.

> **Why not use internal DB IDs?**
> Internal IDs (integer primary keys) are auto-incremented per environment. The same record has `id=5` in dev but `id=105` in prod. The export payload carries stable external keys; the import layer translates them to local IDs at runtime.

### Two-step import workflow

```
Client                              Server
  |                                    |
  |--- POST /import/check ------------>|  validate + dry-run
  |<-- ImportCheckResponse ------------|  includes original_payload
  |                                    |
  | (user reviews warnings)            |
  |                                    |
  |--- POST /import/execute ---------->|  full transactional upsert
  |<-- ImportExecuteResponse ----------|
```

The client sends the **entire `ImportCheckResponse`** to `/execute`. The server re-runs all resolution logic from the embedded `original_payload` — it does not trust any status or ID values the client may have modified.

### Permissions required

All migration endpoints require an **admin JWT**. Non-admin requests receive `403 Forbidden`.

## 2. API Endpoints

| Method | Path | Description |
|---|---|---|
| `GET` | `/bpm/form/:id/export` | Export a form |
| `GET` | `/bpm/workflow/:id/export` | Export a workflow (bundles bound form) |
| `POST` | `/bpm/import/check` | Validate export payload, check dependencies |
| `POST` | `/bpm/import/execute` | Execute transactional import |

All routes are prefixed with `/bpm`. Swagger UI: `http://localhost:3000/bpm/openapi`.

## 3. Export JSON Format

### Common container (wraps every export)

```json
{
  "protocol_version": "1.0",
  "exported_at": "2026-04-07T10:00:00.000Z",
  "exported_by": "user_code_abc",
  "type": "FORM",
  "payload": { ... }
}
```

`type` is `"FORM"` or `"WORKFLOW"`.

---

### Form payload

```json
{
  "public_id": "Xk7mN3pQr8sT",
  "is_template": false,
  "latest_revision": {
    "public_id": "Yt9bL2vWq4nE",
    "name": "Leave Request v1",
    "description": "Standard leave request form",
    "form_schema": { ... },
    "options": {
      "can_withdraw": true,
      "can_copy": true,
      "can_draft": false,
      "can_delegate": false
    }
  },
  "dependencies": {
    "tags": [
      { "name": "HR", "description": "Human Resources", "color": "#FF0000" }
    ],
    "permissions": [
      { "grantee_type": "EVERYONE", "grantee_value": "", "grantee_code": null, "action": "VIEW" },
      { "grantee_type": "USER", "grantee_value": "42", "grantee_code": "alice_code", "action": "USE" }
    ],
    "validations": [
      {
        "source_id": 10,
        "public_id": "DEVQe4iS7tYo2uA",
        "name": "PhoneNumberCheck",
        "validation_type": "CODE",
        "validation_code": "function validate(v) { return /^\\d+$/.test(v); }",
        "error_message": "Must be digits only",
        "components": ["input"]
      }
    ],
    "master_data": [
      { "dataset_name": "Country List" },
      { "dataset_name": "Department Codes" }
    ]
  }
}
```

**Notes:**
- `options` may be `null` if the form has no revision options configured.
- `validations` contains **only** the validators actually referenced by form fields, via either:
  - Array format: `registryValidators[].validatorId` (standard)
  - Flat format: `validator.validatorId` (schema extension, treated identically)
- `master_data` is extracted from:
  - `getMasterData("...")` expressions in the form schema
  - Dynamic dropdown datasources: `datasource.table.tableKey` (top-level fields) and `datasourceType.table.tableKey` (grid columns)

  It is deduplicated — each dataset name appears at most once.
- `source_id` in each validation entry is the internal DB ID from the source environment. It is informational only; the import process ignores it.
- `permissions` carries all access-control grants so they are replicated to the target environment. Three fields are present on each entry:
  - `grantee_value`: source-environment internal database ID of the grantee (integer as string). Remapped to the target environment's ID during execute.
  - `grantee_code`: portable, environment-independent identifier — user `code` or org-unit `code`. `null` for `EVERYONE`. Used for cross-environment grantee lookup.
  - `grantee_type`: `USER`, `ORG_UNIT`, or `EVERYONE`.

---

### Workflow payload

```json
{
  "public_id": "Hc5jD8fGu1xZ",
  "latest_revision": {
    "public_id": "Pz6aK0wBm9rV",
    "name": "Manager Approval Flow",
    "description": null,
    "flow_definition": { ... },
    "options": {
      "reuse_prior_approvals": true
    }
  },
  "binding": {
    "target_form_public_id": "Xk7mN3pQr8sT",
    "bundled_form": { ... }
  },
  "dependencies": {
    "tags": [
      { "name": "Approval", "description": null, "color": "#00FF00" }
    ],
    "permissions": [
      { "grantee_type": "ORG_UNIT", "grantee_value": "101", "grantee_code": "HR_DEPT", "action": "USE" }
    ],
    "users": [
      { "source_id": 42, "code": "CEO_USER_CODE", "name": "Jane CEO", "email": "jane@example.com" }
    ],
    "org_units": [
      { "source_id": 101, "code": "HR_DEPT", "name": "Human Resources", "type": "ORG_UNIT" },
      { "source_id": 55, "code": "APPROVER_ROLE", "name": "Approver Role", "type": "ROLE" }
    ]
  }
}
```

**Notes:**
- `binding.bundled_form` is always included when a form is bound to the workflow. The import will upsert the bundled form first, then the workflow.
- `dependencies.users` and `dependencies.org_units` contain only the entries referenced inside `flow_definition` approver nodes. These are **blocking** dependencies — the import will be blocked if any are missing in the target.
- `source_id` in users and org_units is the source DB ID, used as the key for ID-translation during import.

## 4. Import Check Response Format

```json
{
  "can_proceed": true,
  "summary": {
    "entity_exists": true,
    "action": "UPDATE_REVISION",
    "revision_diff": true
  },
  "dependencies_check": {
    "tags": [
      { "name": "HR", "status": "EXISTS", "severity": "INFO" },
      { "name": "NewTag", "status": "MISSING", "severity": "WARNING" }
    ],
    "validations": [
      {
        "name": "PhoneNumberCheck",
        "status": "EXISTS",
        "severity": "INFO",
        "source": {
          "public_id": "DEVQe4iS7tYo2uA",
          "validation_type": "CODE",
          "validation_code": "function validate(v) { return /^\\d+$/.test(v); }",
          "error_message": "Must be digits only"
        },
        "target": {
          "public_id": "PRDFm3cR6nKj0wB",
          "validation_type": "CODE",
          "validation_code": "function validate(v) { return /^\\d+$/.test(v); }",
          "error_message": "Must be digits only"
        }
      },
      {
        "name": "OtherCheck",
        "status": "MISSING",
        "severity": "INFO",
        "source": {
          "public_id": "DEVNp2wX9vKm4rQ",
          "validation_type": "CODE",
          "validation_code": "function validate(v) { return v.length > 0; }",
          "error_message": "Cannot be empty"
        },
        "target": null
      }
    ],
    "master_data": [
      { "dataset_name": "Country List", "status": "EXISTS", "severity": "INFO" },
      { "dataset_name": "Unknown Dataset", "status": "MISSING", "severity": "WARNING" }
    ],
    "org_units": [
      { "code": "HR_DEPT", "status": "EXISTS", "severity": "INFO" },
      { "code": "GHOST_DEPT", "status": "MISSING", "severity": "BLOCKING" }
    ],
    "users": [
      { "code": "CEO_USER_CODE", "status": "EXISTS", "severity": "INFO" }
    ],
    "permissions": [
      { "grantee_type": "USER", "grantee_code": "alice_code", "action": "USE", "status": "EXISTS", "severity": "INFO" },
      { "grantee_type": "ORG_UNIT", "grantee_code": "GHOST_DEPT", "action": "VIEW", "status": "MISSING", "severity": "WARNING" },
      { "grantee_type": "EVERYONE", "grantee_code": null, "action": "VIEW", "status": "EXISTS", "severity": "INFO" }
    ],
    "related_form": {
      "public_id": "Xk7mN3pQr8sT",
      "status": "IN_BUNDLE",
      "severity": "INFO"
    }
  },
  "original_payload": { ... }
}
```

### Validator `source` and `target` fields

Each entry in `dependencies_check.validations` now carries the full details of both the source validator (from the export) and the matched target validator (from the target DB):

| Field | When present | Content |
|---|---|---|
| `source` | Always | `public_id`, `validation_type`, `validation_code`, `error_message` from the export payload |
| `target` | `status === EXISTS` only | Same fields from the matched validator in the target environment |
| `target` | `status === MISSING` | `null` |

**Why this matters:** Validators are matched by `name` across environments. A name match does not guarantee the validator logic is identical. By comparing `source` and `target`, the frontend can surface meaningful differences:

- `source.public_id !== target.public_id` — the validator exists in both environments but was created independently. The IDs will be remapped automatically during execute (see §7), but the code may differ.
- `source.validation_code !== target.validation_code` — the logic differs between environments. The import will use the **target's existing code**, not overwrite it. This may warrant manual review.

**Recommended frontend behaviour:** when `status === EXISTS` and either `public_id` or `validation_code` differs between `source` and `target`, display a visual indicator (e.g. a warning badge) so the user can confirm before proceeding to execute.

### `dependencies_check.permissions` array

One entry per permission row in the export payload. Reports whether each grantee exists in the target environment.

| Field | Meaning |
|---|---|
| `grantee_type` | `USER`, `ORG_UNIT`, or `EVERYONE` |
| `grantee_code` | Portable grantee identifier (user code / org-unit code); `null` for `EVERYONE` |
| `action` | The permission action (`VIEW`, `USE`, `MANAGE`) |
| `status` | `EXISTS` if the grantee was found in the target; `MISSING` if not |
| `severity` | `INFO` if found; `WARNING` if not |

Missing permission grantees are **never blocking** — `can_proceed` is never set to `false` by permission checks. During execute, rows whose grantee cannot be resolved are silently skipped.

### `summary.action` values

| Value | Meaning |
|---|---|
| `CREATE` | Entity does not exist in target — will be created |
| `UPDATE_REVISION` | Entity exists but revision differs — a new revision will be added |
| `NO_CHANGE` | Entity exists and revision `public_id` matches — nothing to do |

### `summary.revision_diff`

`true` if the export revision `public_id` is different from the latest revision in the target. `false` if they match (action will be `NO_CHANGE`).

### `dependencies_check` status values

| Value | Meaning |
|---|---|
| `EXISTS` | Found in target environment |
| `MISSING` | Not found — see severity for impact |
| `IN_BUNDLE` | The form is included in the workflow export and will be created during import |

### `original_payload`

The complete export container is echoed back. Pass this entire `ImportCheckResponse` object to `/import/execute` unchanged.

## 6. Dependency Resolution Behaviour

### Severity table

| Dependency | Status | Severity | `can_proceed` | What execute does |
|---|---|---|---|---|
| Tag | `EXISTS` | `INFO` | unaffected | Uses existing tag |
| Tag | `MISSING` | `WARNING` | stays `true` | Auto-creates the tag |
| Validation | `EXISTS` | `INFO` | unaffected | Uses existing validator |
| Validation | `MISSING` | `INFO` | stays `true` | Auto-creates the validator + component bindings |
| Master Data | `EXISTS` | `INFO` | unaffected | No action (data must already exist) |
| Master Data | `MISSING` | `WARNING` | stays `true` | **No action** — dataset must be created manually before executing the form |
| User | `EXISTS` | `INFO` | unaffected | Maps source ID → target ID |
| User | `MISSING` | `BLOCKING` | set to `false` | — |
| Org Unit / Role | `EXISTS` | `INFO` | unaffected | Maps source ID → target ID |
| Org Unit / Role | `MISSING` | `BLOCKING` | set to `false` | — |
| Related Form | `EXISTS` or `IN_BUNDLE` | `INFO` | unaffected | Uses existing / imports bundled |
| Related Form | `MISSING` (no bundle) | `BLOCKING` | set to `false` | — |
| Permission (grantee) | `EXISTS` | `INFO` | unaffected | Remaps `grantee_value` to target ID; writes to FormPermission / WorkflowPermission |
| Permission (grantee) | `MISSING` | `WARNING` | stays `true` | **Silently skipped** — permission row not created |

### What "auto-creates" means for validators

When a validation rule is missing in the target, execute creates it with:
- The `name`, `validation_type`, `validation_code`, and `error_message` from the export.
- Component type bindings from `components[]`.
- Description set to `"Imported from <exported_by>"`.
- A **freshly generated `public_id`** — not the `source_id` or the source `public_id` from the payload.

After creation the form schema's `validatorId` references are rewritten to match the new target `public_id` (see §7).

### What "auto-creates" means for tags

Missing tags are created with `name`, `description`, and `color` from the export. A new internal DB ID is assigned.

### Master data behaviour

Master data datasets must be created manually in the target before importing a form that references them. The import will proceed even if they are missing (severity is `WARNING`, not `BLOCKING`), but the form will fail at runtime when trying to resolve `getMasterData("...")` expressions.

### Permission sync behaviour

During execute, the import clears and rewrites all permissions on the target form/workflow using the exported `dependencies.permissions[]` list.

- `EVERYONE` rows are always written.
- `USER` / `ORG_UNIT` rows are looked up by `grantee_code` in the target. If found, `grantee_value` is remapped to the target environment's internal ID and the row is written. If not found, the row is silently skipped.
- Export payloads that lack `grantee_code` on non-EVERYONE rows (produced by older server versions) will have those rows skipped to prevent phantom permissions pointing at wrong users.

## 7. Cross-Environment ID Handling

### The problem

Form schemas store validator references as `validatorId` strings, which are the `public_id` values of validation registry entries. When a validator was created independently in the target environment (not via import), its `public_id` carries the target-env prefix:

```
Source (DEV) form schema:  validatorId = "DEVQe4iS7tYo2uA"
Target (PRD) validator:    public_id   = "PRDFm3cR6nKj0wB"   ← same name, different ID
```

If the form schema is saved verbatim, the `validatorId` still references the dev `public_id`, which does not exist in prod — the form will fail at runtime.

> **When this does NOT happen:** if the target validator was itself created by a prior migration from the same source, it will have inherited the source `public_id` (`DEVQe4iS7tYo2uA`) and no remapping is needed. The remap is a safety net for the independent-creation case.

### The fix (implemented)

During execute, after resolving or creating each validator, the service builds a translation map:

```
source public_id       →  target public_id
"DEVQe4iS7tYo2uA"     →  "PRDFm3cR6nKj0wB"
```

Before saving the form revision, the schema is deep-cloned and every validator ID reference (`registryValidators[].validatorId` and the flat `validator.validatorId` format) is rewritten using this map. The saved schema always contains target-environment `public_id` values.

This is transparent — the frontend and QA do not need to do anything special. The check response's `source.public_id` vs `target.public_id` comparison (§4) gives advance visibility into when a remap will occur.

### Tags

Tags are not embedded in the schema JSON. They are stored in a separate join table. During import, resolved tag IDs are collected and written via `updateTags()`, so there is no ID-leak issue.

### Master Data

Master data is referenced by **dataset name** inside expression strings (`getMasterData("Country List")`), not by ID. Names are environment-independent, so no remapping is needed.

### Permissions

Permission `grantee_value` stores the source-environment internal ID of the grantee. During export, a `grantee_code` (user code or org-unit code) is resolved and stored alongside it.

During execute, each `grantee_code` is looked up in the target environment to obtain the target-env ID, then `grantee_value` is rewritten before the permission row is saved. This is analogous to the validator `public_id` remapping.

```
Source (DEV) payload:  grantee_value="42",  grantee_code="alice"
Target (PRD) lookup:   alice → user id 156
Written permission:    grantee_value="156"

Source (DEV) payload:  grantee_value="42",  grantee_code="bob"
Target (PRD) lookup:   bob not found
Written permission:    (skipped)
```

### Validator ID remapping — flat format

The validator ID translation map covers both schema formats:

- Array format: `registryValidators[].validatorId` — standard field
- Flat format: `validator.validatorId` — used by some component types as a schema extension

Both are rewritten in the same pass using the same `sourcePublicId → targetPublicId` map.

## 8. Workflow Import: ID Transformation

Users and Org Units are referenced inside the `flow_definition` JSON by their internal DB IDs (e.g., `user_ids: [42]`, `org_unit_id: 101`). During execute the server:

1. Looks up each exported `code` in the target environment to find the target DB ID.
2. Builds two maps: `sourceUserId → targetUserId` and `sourceOrgUnitId → targetOrgUnitId`.
3. Runs `FlowDefinitionTransformer` which walks the typed node graph and replaces every approver config ID using the maps.

The transformation covers these approver types:

| Approver Type | Field replaced |
|---|---|
| `SPECIFIC_USERS` | `config.user_ids[]` |
| `SPECIFIC_USER_REPORTING_LINE` | `config.user_id` |
| `DEPARTMENT_HEAD` | `config.org_unit_id` |
| `ROLE` | `config.role_id` (mapped via org_unit table) |

If a user or org unit is not found during execute, the transaction is rolled back and a 500 error is returned. This cannot happen if `/check` returned `can_proceed: true` — unless the data was deleted between the check and execute calls.

## 9. Error Conditions

### Soft-deleted target entity

If the form or workflow identified by `public_id` exists in the target but has been soft-deleted (`is_active: false`), check returns:

```json
{
  "can_proceed": false,
  "summary": {
    "entity_exists": true,
    "action": "NO_CHANGE",
    "revision_diff": false,
    "error": "Target form 'Xk7mN3pQr8sT' exists but is deleted. Please restore it or change the ID."
  }
}
```

The `error` field is present only in this case.

### Unsupported protocol version

If `protocol_version` is not `"1.0"`, both check and execute throw a `400 Bad Request`.

### Missing blocking dependencies

`can_proceed: false` is set by check. Execute will still attempt to run (it does not re-check `can_proceed`), but it will throw when trying to resolve a missing user or org unit during ID mapping, rolling back the transaction.

## 10. Bulk Import API

`POST /bpm/import/bulk` accepts a batch of users, org units, and memberships. This is for synchronizing master HR data from an external system (e.g., HRMS), not for migrating forms/workflows.

### Request body

```json
{
  "users": [
    {
      "code": "EMP001",
      "name": "John Doe",
      "email": "john@example.com",
      "sub": "auth0|abc123",
      "jobGrade": 3,
      "defaultOrgCode": "DEPT001",
      "isDeleted": false
    }
  ],
  "orgUnits": [
    {
      "code": "DEPT001",
      "name": "Engineering",
      "type": "ORG_UNIT",
      "parentCode": "COMPANY",
      "isDeleted": false
    }
  ],
  "memberships": [
    {
      "userCode": "EMP001",
      "orgUnitCode": "DEPT001",
      "assignType": "USER",
      "startDate": "2024-01-01T00:00:00.000Z",
      "endDate": "2099-12-31T00:00:00.000Z",
      "note": "Initial assignment",
      "isDeleted": false
    }
  ]
}
```

### Processing order

1. **Org Units** — upsert by `code`, restore if soft-deleted, create parent links.
2. **Users** — upsert by `code`, restore if soft-deleted.
3. **Memberships** — upsert by `(userId, orgUnitId, startDate)`. Overlap detection is enforced.
4. **Default Org Preferences** — updated after memberships are resolved.

### Soft-delete behaviour

Setting `isDeleted: true` on a user or org unit marks it as deleted in the target. For memberships, `isDeleted: true` sets the `end_date` to now, effectively ending the membership.

### Atomicity

The entire bulk import runs in a single `prisma.$transaction`. A failure at any record (including overlap detection) rolls back all changes and returns a `400 Bad Request` with the error message identifying the failing index:

```
Failed at OrgMembership index 2 (User: EMP001, Org: DEPT001): [Overlap Detected] ...
```

### `orgUnitType` values

`ORG_UNIT` | `ROLE` | `JOB_GRADE` | `POSITION`

### `assignType` values

`USER` | `JOB_GRADE` | `ROLE`

### Response

`204 No Content` on success.

## 11. Step-by-Step Integration Guide

### Migrating a form

```
1. GET  /bpm/form/{formPublicId}/export
        → save the export container JSON

2. POST /bpm/import/check
        body: <export container>
        → review dependencies_check
          - WARNING items (tags, master data): note which need manual setup
          - BLOCKING items: fix before continuing (should not occur for forms)
          - can_proceed must be true

3. POST /bpm/import/execute
        body: <the full ImportCheckResponse from step 2>
        → on 201: import succeeded
        → latest_revision_public_id confirms what was written
```

### Migrating a workflow

Same as form, but use `GET /bpm/workflow/{workflowPublicId}/export`. The workflow export automatically bundles the bound form. Ensure users and org units referenced in the workflow exist in the target before running check (they cause BLOCKING failures).

### Handling `WARNING` dependencies

| Warning | What to do |
|---|---|
| Tag `MISSING` | Nothing — import auto-creates it. |
| Master Data `MISSING` | Create the dataset manually in the target before running execute, or accept that form fields dependent on it will error at runtime. |
| Validation `MISSING` | Nothing — import auto-creates it. |

### Handling `NO_CHANGE`

If check returns `action: "NO_CHANGE"`, the revision already exists in the target. Executing the import is safe but will overwrite the revision with the exported content and re-sync all dependencies. If no update is desired, skip execute.
